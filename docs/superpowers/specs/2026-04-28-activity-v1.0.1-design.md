## 1. 背景と動機

v0.7.0 で Activity diagram の 6 node kinds + 4 制御構造 + swimlane + note が完成、v0.7.1 で hover-insert (action のみ) を追加した。しかし**iterative-workflow editing** の核心がまだ欠落しており、ユーザから「ツールとして破綻している」との指摘を受けた:

> アクションの挿入がアクションのみなので途中で If 文や Loop を表現したい場合、後段のすべての内容を削除しないと追加できない。

具体的破綻パターン:

1. **新規制御構造の途中追加が不可** — 既存 `:A; :B; :C;` の途中に if/while/repeat/fork を入れたい場合、後段全消し → 再入力が必要
2. **既存制御構造の branch 拡張が不可** — 既存 `if (cond?) then :X; endif` に else 分岐を後付けしたい場合、構造ごと delete → 再挿入が必要
3. **既存制御構造の branch 整理が不可** — 不要な elseif / else / fork again を消すには構造ごと消すしかない

これは `feedback_dsl_editor_workflow.md` にメモした「DSL/図形エディタは作成だけでなく途中挿入/並び替え/注釈/制御構造の修正フローを必ず仕様に含める」原則の典型欠落。

v1.0.1 のゴール:

1. **mid-insert kind 拡張** — tail-add と同等の全 7 kind (`action / if / while / repeat / fork / swimlane / note`) を任意位置に挿入可能
2. **composite 内挿入対応** — if / while / repeat / fork の body 内にもホバー位置に応じて挿入可能 (indent 自動継承)
3. **既存制御構造の branch 追加** — 既存 if に `+ elseif 追加` `+ else 追加`、既存 fork に `+ fork again 追加`
4. **既存制御構造の branch 個別削除** — 単一 elseif / else / fork again のみ削除、構造本体は保持

## 2. 適用範囲 (in / out)

### v1.0.1 で達成する (in)

#### A. mid-insert 全 kind (任意位置)

| kind | modal フィールド | 生成 DSL (placeholder付) |
|---|---|---|
| `action` | `text` | `:text;` (1 行) |
| `if` | `cond / thenLabel / elseLabel` | `if (cond) then (thenLabel)\n  :;\nelse (elseLabel)\n  :;\nendif` |
| `while` | `cond / label` | `while (cond) is (label)\n  :;\nendwhile` |
| `repeat` | `cond / label` | `repeat\n  :;\nrepeat while (cond) is (label)` |
| `fork` | `branchCount` (≥2) | `fork\n  :;\nfork again\n  :;\nend fork` (branchCount 数の placeholder) |
| `swimlane` | `name` | `\|name\|` (1 行) |
| `note` | `position (left/right) / text` | `note right\n  text\nend note` または inline `note right :text` (target は省略、PlantUML が直前 action に自動 attach) |

- 挿入位置: ホバー Y 座標から `resolveInsertLine` で line/position 取得 (composite 内の場合は内部 line が返る)
- indent: 既存 v0.7.1 ロジック (target line 継承) + 新規制御構造の内部行は **継承 indent + 2 spaces**
- placeholder action は空テキスト `:;` (PlantUML legal、ユーザが後で edit)

#### B. composite 内挿入

- 既存 `if (a?) then\n  :X;\nendif` の `:X;` 周辺ホバー → if/while/etc を挿入すると `:X;` の indent (2 spaces) を継承して入れ子構造完成
- 例: `:X;` の前に if 挿入 → `if (a?) then\n  if (newCond?) then\n    :;\n  endif\n  :X;\nendif`
- 制限: composite 内挿入は indent 継承のみ。**コンテキストに応じた kind 制限は行わない** (例: fork の中に if 入れるのも許可、PlantUML が許す限り通す)

#### C. 既存制御構造への branch 追加

- **対象 element**: `if` / `fork` のみ (while / repeat は branch 概念なし)
- **if への追加**:
  - `+ elseif 追加` → endif (or 既存 else) の直前に elseif + placeholder action を挿入
  - `+ else 追加` → endif の直前に else + placeholder action を挿入。**既存 else があれば button disabled**
  - 入力: cond (elseif のみ) / label は inline prompt() で取得 (modal 化は次スプリント)
- **fork への追加**:
  - `+ fork again 追加` → end fork の直前に fork again + placeholder を挿入
  - 入力なし (即座に追加)

#### D. 既存制御構造の branch 個別削除

- **対象 branch 行**: `elseif` / `else` / `fork again` 各行
- **対象外**: `if` 開始行、`endif`、`fork` 開始行、`end fork` (これらは構造ごと削除でしか消せない)
- UI: `_renderControlEdit` の各 branch 行に `✕` ボタン追加 (confirm dialog つき)
- 動作: 該当 branch line から次 branch line (or endif/end fork) の直前まで削除
- 例: `if (a) then :X; elseif (b) :Y; else :Z; endif` の elseif 行 ✕ → `if (a) then :X; else :Z; endif` (elseif 配下の `:Y;` も一緒に削除、前後の `:X;` / else 分岐 `:Z;` は影響なし)
- **後続 branch との結合は行わない** — 削除範囲は elseif 行から次の branch line (`else` or `endif`) の直前まで。隣接する then / else 部分はマージされない

### v1.0.1 では やらない (out)

- **既存ノードを if/while で囲む UX** (「wrap with if」操作) — Q2 で却下、別ニーズなので v1.x+ で別途検討
- **branch 順序入れ替え (drag-to-reorder)** — Activity v0.7.x 共通の繰越項目
- **branch 追加時の modal 化** — まず prompt() で出して、必要が顕在化したら modal 化
- **State 図の同種 polish** — v1.0.2 で別 spec
- **Sequence の multi-select connect** — Tier2 横断項目
- **mid-insert kind ごとの validation** (例: fork の中に start を入れるのを禁止) — PlantUML が落とすので UI で先回り validation はしない

## 3. アーキテクチャ

### 3.1 新規 / 拡張 API (`src/modules/activity.js`)

#### 挿入 API (5 個)

```js
// kind ∈ {if, while, repeat, fork} の DSL を生成して line 周辺に挿入
function addControlAtLine(text, lineNum, position, kind, fields) { ... }

// |name| を 1 行挿入
function addSwimlaneAtLine(text, lineNum, position, name) { ... }

// note (left|right) of を 1〜複数行挿入。"of TARGET" は省略 (直前 action へ自動 attach は PlantUML 解釈に任せる)
function addNoteAtLine(text, lineNum, position, fields) { ... }
```

既存 `addActionAtLine(text, lineNum, position, actionText)` は据え置き。

#### branch 操作 API (4 個)

```js
// 既存 if (ifLine) の endif (or 既存 else) の直前に elseif + placeholder を挿入
function addElseifBranch(text, ifLine, condition, label) { ... }

// 既存 if (ifLine) の endif 直前に else + placeholder を挿入。既存 else があれば no-op (false 返す)
function addElseBranch(text, ifLine, label) { ... }

// 既存 fork (forkLine) の end fork 直前に fork again + placeholder を挿入
function addForkBranch(text, forkLine) { ... }

// 単一 branch (elseif | else | fork again) のみ削除。指定 line から次 branch line (or endif/end fork) の直前まで削除
function deleteBranchAt(text, branchLine) { ... }
```

すべてエクスポート (`return { ... }`)。

### 3.2 UI 拡張

#### `showInsertForm(ctx, line, position, kind)` 再構成

現状はアクション専用 1 textarea。tail-add パネル (`#ac-tail-*`) と同じ kind selector + 条件フィールドの構造に再構成する。

modal HTML テンプレート (`act-modal` 内に注入):

```html
<h3>(L{line} の {position}) に挿入</h3>
<select id="act-mod-kind">  <!-- 7 options -->
<div id="act-mod-fields">
  <!-- kind に応じて切替表示 (action/if/while/repeat/fork/swimlane/note) -->
</div>
<button id="act-mod-cancel"> / <button id="act-mod-confirm">
```

`act-mod-kind` の change イベントで `act-mod-fields` を再描画 (innerHTML 切替)。confirm 時に kind に応じて対応 API を呼ぶ。

#### `_renderControlEdit` 拡張

既存 (src/modules/activity.js:1130-1209) に以下を追加:

- **if の場合**:
  - branches リスト各行に `✕ delete` ボタン追加 (kind が elseif / else の行のみ)
  - リスト下部に `+ elseif 追加` ボタン (常時表示)、`+ else 追加` ボタン (既存 else がなければ表示)
- **fork の場合**:
  - branches リストを表示 (現状は count のみ)、各 fork-again 行に `✕ delete`
  - `+ fork again 追加` ボタン

### 3.3 modal HTML 拡張 (`plantuml-assist.html`)

既存 `<div id="act-modal">` 内の content area に新しい form を入れるため、構造は据え置き、content innerHTML を JS から動的注入。**HTML 静的部分の変更は不要**。

### 3.4 indent 戦略

- `addActionAtLine` の `indent = (lines[targetIdx].match(/^(\s*)/) || ['',''])[1]` ロジックを共有ヘルパに抽出 (`_resolveIndent(lines, targetIdx)`)
- 制御構造の内部行: `indent + '  '` (2 spaces)
- branch 追加時: 親 if/fork の indent (= ifLine の indent) を取得し、branch line は `indent`、placeholder は `indent + '  '`

## 4. ユースケーストレース (use-case-driven-ui-spec)

### UC-1: 既存フローの途中に if を追加 (新規構造挿入の典型)
1. ユーザは `:Login;\n:Render UI;` の Activity 図を開いている
2. `:Login;` と `:Render UI;` の間 (空白) にホバー → `+ ここに挿入` ガイド表示
3. クリック → modal 開く (現在は kind=action 固定で開く想定 = 既定値)
4. kind selector で `if` を選択 → modal 内のフィールドが cond/thenLabel/elseLabel に切替
5. cond=`auth?` thenLabel=`yes` elseLabel=`no` を入力 → 確定
6. DSL: `:Login;\nif (auth?) then (yes)\n  :;\nelse (no)\n  :;\nendif\n:Render UI;`
7. プレビュー再描画、新 if が表示される。中の placeholder は後でクリックして編集

### UC-2: 既存 if の中に新たな if を入れ子で追加
1. 既存 `:A;\nif (cond1?) then\n  :B;\nelse\n  :C;\nendif`
2. then-branch の `:B;` 直前にホバー → ガイド表示 (composite 内で)
3. クリック → modal、kind=if、cond=`cond2?` 入力 → 確定
4. DSL: `:A;\nif (cond1?) then\n  if (cond2?) then\n    :;\n  endif\n  :B;\nelse\n  :C;\nendif`
5. indent が 2 spaces / 4 spaces で正しく階層化されている

### UC-3: 既存 if に else を後付け
1. 既存 `if (a?) then\n  :X;\nendif` (else なし)
2. if の polygon (decision) を SVG で click → property panel に if 編集フォーム
3. branches セクションに `+ elseif 追加` `+ else 追加` ボタンが表示
4. `+ else 追加` クリック → label 入力 prompt() → 確定
5. DSL: `if (a?) then\n  :X;\nelse\n  :;\nendif`
6. 既存 else があれば `+ else 追加` ボタンは disabled

### UC-4: 既存 if に elseif を 2 個連続追加
1. 既存 `if (a?) then :X; endif`
2. `+ elseif 追加` → cond=`b?` label=`yes` → DSL: `... elseif (b?) then (yes) :;`
3. もう一度 `+ elseif 追加` → cond=`c?` → 直前の elseif の後に挿入
4. 結果: `if (a?) then :X; elseif (b?) then (yes) :; elseif (c?) then (yes) :; endif`

### UC-5: 不要な elseif を 1 つだけ削除
1. 既存 `if (a?) then :X; elseif (b?) :Y; else :Z; endif`
2. property panel の elseif 行の `✕` クリック → confirm
3. DSL: `if (a?) then :X; else :Z; endif` (Y は削除、X/Z 残存)

### UC-6: fork に分岐を追加して削除
1. 既存 `fork :X; fork again :Y; end fork`
2. fork bar (rect) を click → `+ fork again 追加` ボタン
3. クリック → DSL: `fork :X; fork again :Y; fork again :; end fork`
4. 新分岐の `:` placeholder を後で編集
5. 不要になった分岐の `✕` で個別削除

### UC-7: while ループの中に action を mid-insert
1. 既存 `while (more?) is (yes)\n  :process;\nendwhile`
2. `:process;` 直後にホバー → クリック → kind=action、text=`log result` → 確定
3. DSL: `while (more?) is (yes)\n  :process;\n  :log result;\nendwhile`
4. v0.7.1 で実装済の挙動だが、composite 内であることを再確認するテスト

### UC-8: swimlane を mid-insert
1. 既存 `:A;\n:B;`
2. `:A;` と `:B;` の間でクリック → kind=swimlane、name=`Backend` → 確定
3. DSL: `:A;\n|Backend|\n:B;`
4. プレビュー再描画、`:B;` が Backend swimlane に移動

### UC-9: note を mid-insert
1. 既存 `:A;\n:B;`
2. `:A;` と `:B;` の間でクリック → kind=note、position=right、text=`important` → 確定
3. DSL: `:A;\nnote right\n  important\nend note\n:B;`
4. PlantUML が直前 action に attach、:A; の右に表示

## 7. リスクと回避

| リスク | 影響 | 回避策 |
|---|---|---|
| `resolveInsertLine` が composite 境界 (`endif`, `endwhile`) 行を返した場合の indent 誤継承 | 入れ子崩壊 | `addControlAtLine` で「対象行が `endif/endwhile/end fork/else/elseif/repeat while/fork again` の場合は前行の indent を採用」のフォールバックを実装 |
| 既存 elseif の中間に新 elseif を入れたい (順序変更) | UX gap | v1.0.1 では「末尾追加」のみ対応。reorder は別 sprint |
| placeholder `:;` で PlantUML が描画失敗するケース | プレビュー破損 | 検証: PlantUML 公式 jar で `:;` がエラーにならないことを事前確認、ダメな場合は `:TODO;` にフォールバック |
| modal kind change で残ったフィールド値が意図せず確定される | 誤挿入 | kind change で `act-mod-fields` を innerHTML 全置換 (state はクリア) |
| branch 追加時の prompt() で空入力 | 不正 DSL | cond 空なら no-op (button click が成立しない)、label 空なら default (`yes`) |

## 9. v1.0.2+ への明示繰越

- **State 図の同種 polish** — mid-insert (state, transition, note) と composite-state 内挿入、本 spec と同型の問題を解消
- **既存ノードを if/while で囲む UX** — wrap-existing operation
- **branch 順序入れ替え** — drag-to-reorder (Activity と Class member 共通設計)
- **branch 追加 modal 化** — prompt() ベースから dedicated modal へ
- **mid-insert 後の placeholder への直接フォーカス** — 挿入完了後に新 placeholder action を property panel で自動 expand
