# Activity v1.0.2 Design (Branch-aware insertion)

- **作成日**: 2026-04-29
- **対象**: PlantUMLAssist v1.0.2 — v1.0.1 mid-insert の critical UX bug fix
- **base ブランチ**: `feat/activity-v1.0.1` (PR #12 merge 後は master 相当)
- **次ブランチ**: `feat/activity-v1.0.2`
- **親 spec**: `docs/superpowers/specs/2026-04-28-activity-v1.0.1-design.md`
- **適用 skill**: `dsl-editor-spec-checklist`, `use-case-driven-ui-spec`

---

## 1. 背景と動機

v1.0.1 で全 7 kind の mid-insert + composite 内挿入を実装した。しかし、`resolveInsertLine(overlayEl, y)` (src/modules/activity.js:738) は **Y 座標のみで挿入位置を解決し X 座標を完全無視** している。

PlantUML Activity は if/fork の分岐を**水平に並べてレンダリング** する:

```
        ◇decision
       /         \
   :LEFT;       :RIGHT;
       \         /
        merge
```

このレイアウトで、ユーザが else 側 (右カラム) の空白をクリックしても、Y が同じ then 側 (左カラム) の rect が DOM 文書順で先に出現する場合、then 側の最近接 rect が選ばれて意図しない branch に挿入される。

**ユーザ指摘** (2026-04-29):

> ここに挿入が行レベルで管理されるので分岐があるとどちらの分岐に挿入されるかわかりません。

これは v1.0.1 の mid-insert 機能の信頼性を損なう critical UX bug。 トップレベルの垂直フローでは表面化しないが、composite を含む実用 DSL では頻発する。

v1.0.2 のゴール:

1. **X 座標を考慮した最近接 rect 解決** — クリック位置 (X, Y) に最もユークリッド距離が近い overlay rect を選ぶ
2. **Hover ガイドを resolved rect の X 範囲に限定** — 全幅の dashed line を、解決された rect のカラム幅に短縮し「どの列に入るか」を視覚化

## 2. 適用範囲 (in / out)

### v1.0.2 で達成する (in)

#### A. X-aware `resolveInsertLine`

**signature 変更**: `resolveInsertLine(overlayEl, y)` → `resolveInsertLine(overlayEl, x, y)`

**アルゴリズム**:

```
1. overlay rects (data-type ∈ {action, decision, start, stop, end, fork}) を全て取得
2. rects.length === 0 なら null
3. 各 rect の中心 (cx, cy) を計算: cx = rect.x + rect.width / 2, cy = rect.y + rect.height / 2
4. ユークリッド距離 d = sqrt((x - cx)^2 + (y - cy)^2) を計算、最小の rect を best として選ぶ
5. position = (y < best.cy) ? 'before' : 'after'
6. return { line: best.line, position: position }
```

**期待される挙動**:
- トップレベルの垂直フロー: 全 rect の cx が同じため X 距離は 0、Y 距離が支配的 → 従来挙動と一致
- 水平分岐 composite: 同 Y の rect が複数ある場合、クリック X に近い rect が勝つ → 視覚と一致
- 完全に外れたクリック (rect から遠く離れた位置): 全 rect から距離計算 → 最も「近い感じ」の rect が選ばれる (旧来の「Y で walk」とは挙動差があるが UX 上自然)

#### B. Hover ガイドを resolved rect カラムに限定

`#hover-layer .hover-guide` の dashed line span を、解決された rect の `[rect.x - PADDING, rect.x + rect.width + PADDING]` に制限。 PADDING は 8-12px 目安。

これにより:
- 単一カラム (top-level): ガイド線は普通の長さ
- 分岐内: ガイド線は該当カラムのみ。別カラムには伸びない → 「else 列に入る」「then 列に入る」が一目で分かる

#### C. 呼び出し元 (caller) の改修

`src/app.js` (or wherever overlay-layer hover/click events are wired) の以下を改修:
- mousemove handler: hover X/Y の両方を取得して `resolveInsertLine(overlayEl, x, y)` に渡す
- click handler: クリック X/Y の両方を取得して同様に渡す

ガイド描画ロジック (どこで `.hover-guide` を `<line>` として作っているか) も合わせて X 範囲指定に変更。

### v1.0.2 では やらない (out)

- **buildOverlay が composite 内 action の overlay rect を emit しない件** (v1.0.1 UC-2 の skip 原因) — 別バグ、 v1.0.3+ で別 sprint。 X-aware にしても overlay rect が無ければ branch 解決のしようがない、ただし top-level の分岐 (then/else 直下のみ) は v1.0.2 でも改善される
- **branch 名 (then/else/fork-N) のテキスト表示** — `.hover-guide` 横に「else 分岐」と表示する案。視覚 cue (X 範囲限定) で十分なため YAGNI
- **weighted distance** (X/Y で重み変更) — pure euclidean で十分動作するため YAGNI
- **State 図の同種 polish** — v1.0.3 別 spec
- **drag-to-connect / drag-to-reorder** — Tier2 横断項目

## 3. アーキテクチャ

### 3.1 `src/modules/activity.js` の変更

`resolveInsertLine` の置換 (line 738 付近):

```javascript
// Map a click/hover position (in SVG/overlay coordinates) to the nearest
// rect by Euclidean distance, then determine before/after by Y relative to
// the rect center. X is honored for branch disambiguation in if/fork composites.
function resolveInsertLine(overlayEl, x, y) {
  if (!overlayEl) return null;
  var rects = overlayEl.querySelectorAll(
    'rect[data-type="action"], rect[data-type="decision"], rect[data-type="start"],' +
    'rect[data-type="stop"], rect[data-type="end"], rect[data-type="fork"]'
  );
  if (rects.length === 0) return null;
  var best = null;
  var bestDist = Infinity;
  Array.prototype.forEach.call(rects, function(r) {
    var rx = parseFloat(r.getAttribute('x'));
    var ry = parseFloat(r.getAttribute('y'));
    var rw = parseFloat(r.getAttribute('width'));
    var rh = parseFloat(r.getAttribute('height'));
    var cx = rx + rw / 2;
    var cy = ry + rh / 2;
    var dx = x - cx;
    var dy = y - cy;
    var d = Math.sqrt(dx * dx + dy * dy);
    if (d < bestDist) {
      bestDist = d;
      best = {
        line: parseInt(r.getAttribute('data-line'), 10),
        cx: cx,
        cy: cy,
        rx: rx,
        rw: rw,
      };
    }
  });
  if (!best) return null;
  return {
    line: best.line,
    position: y < best.cy ? 'before' : 'after',
    // for hover-guide rendering: caller can use these to constrain the guide span
    rectX: best.rx,
    rectWidth: best.rw,
  };
}
```

**互換性**: 戻り値の `{line, position}` は同じ。 追加で `rectX`, `rectWidth` を返す (caller がガイド描画に使用)。 既存 caller が rectX/rectWidth を無視しても動作は壊れない。

### 3.2 `src/app.js` の caller 改修

現状想定 (推定):

```javascript
overlayEl.addEventListener('mousemove', function(ev) {
  var rect = overlayEl.getBoundingClientRect();
  var y = ev.clientY - rect.top;
  var resolved = currentModule.resolveInsertLine(overlayEl, y);
  // draw hover guide ...
});
```

改修後:

```javascript
overlayEl.addEventListener('mousemove', function(ev) {
  var rect = overlayEl.getBoundingClientRect();
  var x = ev.clientX - rect.left;
  var y = ev.clientY - rect.top;
  var resolved = currentModule.resolveInsertLine(overlayEl, x, y);
  // draw hover guide using resolved.rectX / resolved.rectWidth
});
```

具体的な X/Y 計算箇所と guide 描画 SVG 作成箇所は実装時に grep で確認。

### 3.3 Hover ガイドの span 制限

ガイド `<line>` 描画箇所で:

```javascript
var PADDING = 10;
var x1 = resolved.rectX - PADDING;
var x2 = resolved.rectX + resolved.rectWidth + PADDING;
guideLine.setAttribute('x1', x1);
guideLine.setAttribute('x2', x2);
guideLine.setAttribute('y1', /* same as before */);
guideLine.setAttribute('y2', /* same as before */);
```

トップレベルでは X 範囲が overlay 全幅相当の rect (decision diamond / action) の幅 + padding になり、視覚上ほとんど現状と同等になる。 分岐内では rect 1 個分の幅 + padding に縮む。

### 3.4 Sequence モジュールへの波及

`src/ui/sequence-overlay.js` も `resolveInsertLine` を export している。 caller (`src/app.js`) が `currentModule.resolveInsertLine` を経由して呼ぶため、 signature を揃える必要がある。

**決定**: Sequence の `resolveInsertLine` も `(overlayEl, x, y)` signature に変更する。 Sequence 側の実装は X を無視 (関数本体の Y のみ参照ロジックを維持) する単純改修で済む。 これにより caller 側は kind 分岐なく `currentModule.resolveInsertLine(overlayEl, x, y)` を呼べる。

Sequence 側の戻り値に `rectX` / `rectWidth` を含めるかは任意 — 含めなければ Sequence のガイドは従来通り全幅描画される (互換維持)。 v1.0.2 では Activity のみガイド span を制限する。

## 4. ユースケーストレース (use-case-driven-ui-spec)

### UC-1: else 分岐末尾への挿入 (本質ケース)

1. ユーザは下の DSL を編集中:
   ```
   :Login;
   if (auth?) then (yes)
     :Render dashboard;
   else (no)
     :Show login form;
   endif
   :Logout;
   ```
2. PlantUML レンダリング: 左に `:Render dashboard;`、右に `:Show login form;` (水平分岐)
3. ユーザは else 側 (右カラム) の `:Show login form;` 直下の空白をクリック
4. **修正前**: Y のみで解決 → DOM 順で先の `:Render dashboard;` (line 3) が選ばれて、then 末尾に挿入される
5. **修正後**: X+Y 距離で右カラムの `:Show login form;` (line 5) が勝つ → else 末尾に挿入される
6. ユーザの視覚的期待と DSL 結果が一致

### UC-2: then 分岐途中への挿入 (基本動作確認)

1. 既存 DSL: 上記同じ
2. ユーザは then 側 (左カラム) の `:Render dashboard;` のすぐ上をクリック
3. 修正前後とも: line 3 (`:Render dashboard;`) の position='before' に解決
4. then 分岐先頭に挿入される (期待通り)

### UC-3: トップレベル垂直フロー (回帰確認)

1. DSL: `:A; :B; :C;` (分岐なし)
2. ユーザは `:B;` の下をクリック
3. 修正前後とも: line 2 (`:B;`) の position='after' → 同じ挙動
4. トップレベルでは X が均一なため Y のみで決まり、回帰なし

### UC-4: Hover ガイドの span 確認

1. DSL: 上記 UC-1 と同じ if/else
2. ユーザが else 側のカラム (右) でホバー
3. **修正前**: ガイド線が overlay 全幅で表示 → どっちの列に入るか不明
4. **修正後**: ガイド線が `:Show login form;` の X 範囲 + 10px padding に縮小 → 「else 列に入る」が視覚的に明確
5. then 側にホバー移動するとガイド線が左カラム幅に切り替わる

### UC-5: 完全に外れたクリック (エッジケース)

1. DSL: 既存図あり
2. ユーザは図の遥か右下 (overlay 外周近く) をクリック
3. 修正後: 全 rect への距離計算で「最も近いと感じる」rect が選ばれる
4. 修正前は「Y > 全 rect midY → 最後の rect の after」だった
5. 挙動差は出るが、「外れた位置でも挿入が成立する」UX は維持

### UC-6: fork 分岐での挿入

1. DSL:
   ```
   fork
     :Path A;
   fork again
     :Path B;
   fork again
     :Path C;
   end fork
   ```
2. レンダリング: 3 列水平に並ぶ
3. ユーザが Path B (中央列) の下をクリック
4. **修正後**: 中央列の `:Path B;` が最近接 → line 4 の after に解決
5. fork-2 (中央分岐) 末尾に挿入される

## 5. テスト方針

### 5.1 Unit テスト (`tests/activity-overlay.test.js` 拡張)

既存の `resolveInsertLine` describe を拡張:

- 既存テスト 4 個 (Y のみ) → signature 変更で `(overlay, x, y)` に書き直し、X=固定値 (0 or rect 中央)
- 新規テスト 4 個:
  1. **X が異なる 2 rect、X が close な方を選ぶ**: rect A at (50, 100), rect B at (200, 100)、click (180, 100) → B が勝つ
  2. **Y が異なる 2 rect、Y が close な方を選ぶ**: rect A at (50, 100), rect B at (50, 200)、click (50, 110) → A が勝つ (回帰確認)
  3. **X+Y 両方違う、ユークリッド最小**: 距離計算が正しいことを確認
  4. **rectX / rectWidth が返り値に含まれる**: hover guide 用フィールドの存在確認

合計 約 4 テスト追加。

### 5.2 E2E テスト (`tests/e2e/activity.spec.js` 拡張)

- **UC-1 (else 分岐挿入)**: DSL 設定 → else 列の rect の下をクリック → modal で text 入力 → confirm → DSL 検証 (else 末尾に挿入されている、then 末尾には影響なし)
  - ※ buildOverlay の composite 内 action 制限が解消されない限り skip フォールバック維持。 ただし、 top-level の 2 分岐 if (placeholder のみ) で再現できるなら test.skip() を外せる。
- **UC-4 (hover ガイド span)**: 分岐内ホバー → guide line の x1/x2 を取得 → resolved rect の X 範囲内に収まっていることを assert

合計 約 2 E2E テスト追加 (UC-1 は skip 維持の可能性あり)。

### 5.3 Visual verification gate

CLAUDE.md 規定に従い、以下を Playwright で実機確認:
1. else 列クリック → モーダル → 確定後の DSL が else 内に挿入される (スクリーンショット保存)
2. hover 時のガイド線 span が左/右で切り替わる (スクリーンショット 2 枚)

## 7. リスクと回避

| リスク | 影響 | 回避策 |
|---|---|---|
| Sequence 用の `resolveInsertLine` も signature 変更が必要、互換性破壊 | Sequence 側回帰 | Sequence の関数も `(overlayEl, x, y)` に変更、x は内部で無視。テストで Sequence E2E が通ることを確認 |
| `rectX` / `rectWidth` を caller が無視した場合、ガイドが新仕様で描画されない | 視覚 cue 不発 (機能は動く) | caller を必ず更新。 fallback として、resolveInsertLine が undefined を返した時はガイドを描画しない |
| 全 rect 計算で per-mousemove のオーバーヘッド | パフォーマンス劣化 | rect 数は通常 < 30、距離計算は O(n)。実用範囲内。気になれば throttle 追加 (現状 mousemove 自体に debounce/throttle がないため、大規模図でのみ問題化) |
| ユークリッド距離だと「Y を圧倒的に重視したい場面」 (細長い縦フロー) で誤判定 | UX 退化 | 実装後に手動確認、必要なら weighted distance (`dy * 1.5`) に切り替え可能なよう距離関数を抽象化 |
