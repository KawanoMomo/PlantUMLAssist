- **作成日**: 2026-04-29
- **対象**: PlantUMLAssist v1.1.0 — State 図の Tier2 機能追加 + UX 改善
- **base ブランチ**: `feat/activity-v1.0.3` (PR #14 merge 後は master 相当)
- **次ブランチ**: `feat/state-v1.1.0`
- **親 spec**: `docs/superpowers/specs/2026-04-28-state-design.md` (v1.0.0)
- **適用 skill**: `dsl-editor-spec-checklist`, `use-case-driven-ui-spec`

---

## 1. 背景と動機

State 図 v1.0.0 は overlay-driven editing で 6 element kinds + transitions + notes が完成。 ただし以下の課題が残っている:

1. **graph-like layout に hover-insert が不適切** — Activity と異なり State はノード位置が縦に並ぶ保証がない。 hoverInsert + showInsertForm が capabilities で `true` のままだが、 ユーザがクリック位置から挿入先を予測できない
2. **transition の追加が面倒** — 既存 state を選択しても「次の遷移を追加」UI がなく、 DSL 直書き or 別 sprint の Multi-select connect (まだ未実装) のみ
3. **composite state の入れ子操作が DSL 直書き必須** — top-level state を別の composite に取り込む / 出す / 解体する操作が UI 経由で不可
4. **transition の source/target を変更できない** — 既存 transition の trigger/guard/action は編集可、 ただし from/to 変更は DSL 直書き必須
5. **entry/exit/do (during) actions の入力 UI なし** — UML 標準の behavior 記述が PlantUML 上は description line 形式で対応するが、 専用フィールドが property panel になく、 ユーザは `STATE : entry / xxx` 構文を覚えて入力する必要がある

ユーザ指摘 (2026-04-29):

> State は縦方向に図形が生成されることが保証されないから、 ここに挿入で追加するのは不適切。
> entry/exit 処理や during 処理をプロパティからそれぞれ記載できるようにして

v1.1.0 のゴール:

1. **不適切な hoverInsert / showInsertForm を State から除去** (i)
2. **selected state の property panel に + Outgoing transition ボタン** + modal で target 選択 (ii)
3. **composite state nesting ops 全 4 種** (a Convert / b Dissolve / c Move-into / d Move-out) (iii)
4. **transition edit form に source/target dropdown** (vi)
5. **state edit form に Behaviors セクション** (entry / do / exit、 StableState 流の UI) (NEW)

## 2. 適用範囲 (in / out)

### v1.1.0 で達成する (in)

#### A. hoverInsert / showInsertForm 除去 (i)

- `src/modules/state.js` `capabilities.hoverInsert: true` → `false`
- `capabilities.showInsertForm: true` → `false`
- `showInsertForm` 関数本体は当面残す (将来 selection-based insert に再利用する可能性) が、 capabilities が false なので app.js から呼ばれない
- `resolveInsertLine` も signature 維持 (v1.0.2 の互換用)、 ただし呼び出されない
- `#st-modal` HTML 要素 (plantuml-assist.html) は据え置き、 表示されないだけ

#### B. + Outgoing transition UI (ii)

- 既存 `_renderStateEdit` (state edit form) に **`+ Outgoing transition`** ボタン追加
- ボタンクリック → modal popup (`#st-modal` 流用 or 新規 `#st-tx-modal`):
  - **Target state**: dropdown (既存 state ID 一覧、 自分自身も含む = self-transition)
  - **Trigger** (任意): text input
  - **Guard** (任意): text input
  - **Action** (任意): text input
  - 確定 → `addTransition(text, fromId, toId, trigger, guard, action)` を呼ぶ (既存 API)
- 新規 transition は DSL の末尾 (現存の transition 群の後ろ) に追加。 `insertBeforeEnd` で `@enduml` 直前に挿入

#### C. composite state nesting ops 全 4 種 (iii)

##### C-a. Convert to composite

- 単純 state を `state X { }` (空) に変換
- 対象 element: `state` (composite-state ではない、 単純 state のみ)
- property panel `_renderStateEdit` に `+ Convert to composite` ボタン (state.kind === 'state' の時のみ表示)
- DSL 変換: `state X` → `state X {\n}`、 `state X as "Label"` → `state X as "Label" {\n}` (label 維持)
- 新規 API: `convertToComposite(text, stateId)` — line を見つけて `{`/`}` 追加

##### C-b. Dissolve composite

- composite state の `{ }` を外し、 子 states / transitions を top-level に出す
- 対象 element: `composite-state` (kind === 'composite-state')
- property panel `_renderCompositeStateEdit` に `✕ Dissolve composite` ボタン (確認 dialog 付き)
- DSL 変換: `state C { ... children ... }` → `... children ...` (C の枠を消し、 内側の states / transitions を 1 階層上に持ち上げる、 indent を 1 段減らす)
- 新規 API: `dissolveComposite(text, compositeId)`
- 子要素なしなら単純 1 行削除

##### C-c. Move state into composite

- top-level state を別の composite の子として移動
- 対象 element: top-level の `state` (composite-state も含む、 ただし循環参照防止のため自分自身および descendants は dropdown 候補から除外)
- property panel `_renderStateEdit` / `_renderCompositeStateEdit` に **`Move into:` dropdown + `Move` ボタン**
- dropdown: 既存の composite-state IDs (移動先候補)、 デフォルト `-- 選択 --`
- DSL 変換: `state X ... \n state C { ... }` → `state C { ... \n state X ...\n }`
- 新規 API: `moveStateIntoComposite(text, stateId, targetCompositeId)`
- 移動対象 state に紐づく外部 transition (top-level に書かれていた `X --> Y`) は **そのまま top-level に残す** (PlantUML は composite 越境 transition を許容)

##### C-d. Move state out of composite

- composite 内 state を top-level に取り出す
- 対象 element: composite-state の子として存在する state
- property panel に **`↑ Move out of [parent]` ボタン** (state.parentId が non-null の時のみ表示)
- DSL 変換: composite 内の state line を取り出し、 元の composite の `{` 内から削除、 top-level (composite の declaration の前 or 後) に追加
- 新規 API: `moveStateOutOfComposite(text, stateId)` — parent の `{ }` 内から該当 state line を切り出し、 composite declaration の直後 (top-level) に再配置
- 紐づく transition の移動規則: B + 同様に top-level に残す

#### D. transition edit form に source/target dropdown (vi)

- 既存 `_renderTransitionEdit` の trigger/guard/action 編集に加えて:
  - **From state**: dropdown (既存 state IDs、 現在の source が selected)
  - **To state**: dropdown (同上、 現在の target が selected)
- 更新ボタンクリック時、 from/to が変更されていれば DSL 上の transition line を再 emit
- 新規 API: `updateTransitionEndpoints(text, lineNum, newFrom, newTo)` — 該当行を `newFrom --> newTo` で書き換え (label 部分は維持)
- pseudo-state (`[*]` initial / final) も dropdown 選択肢に含める

#### E. Behaviors セクション (entry / do / exit) (NEW)

##### UI (StableState 準拠、 mock 確認済)

- `_renderStateEdit` / `_renderCompositeStateEdit` に **`Behaviors` セクション** 追加:
  - **entry**: single-line `<input>` (placeholder: `entry`)
  - **do**: multi-line `<textarea>` (placeholder: `do`、 改行は `\n` escape で 1 行に結合して DSL emit)
  - **exit**: single-line `<input>` (placeholder: `exit`)
- 「更新」ボタンクリックで 3 フィールドを一括反映

##### DSL 仕様

PlantUML の **description line** 形式を採用:
```plantuml
state Driving as "運転中"
Driving : entry / start_engine()
Driving : do / monitor_speed()\nlog_telemetry()
Driving : exit / stop_engine()
```

- 空フィールドは description line を **emit しない** (空行 / 無意味な entry/exit/do を残さない)
- 既存の description line で `entry / xxx` `exit / xxx` `do / xxx` prefix を持つものは parse 時にこの 3 フィールドへ振り分け
- prefix を持たない description line (`Driving : custom note`) は別管理 (state の `description` フィールドとして保持、 v1.1.0 では編集 UI なしで parse → emit のみ)

##### parser 拡張

- `parse()` の state ノードに新フィールド追加: `entry: string|null`, `do: string|null`, `exit: string|null`, `descriptions: string[]`
- description line の regex: `/^\s*(\w+)\s*:\s*(?:(entry|exit|do)\s*\/\s*)?(.+)$/`
  - prefix 一致 → state node の対応フィールドへ
  - prefix なし → `descriptions` 配列へ append
- 多行 do は parse 時に `\n` escape のまま保持、 UI 側でテキストエリア表示時に実際の改行に変換

##### updater API

- `setStateBehavior(text, stateId, kind, value)` — kind ∈ {entry, do, exit}、 該当 description line を update / insert / delete
- value === '' or null → 該当行を削除
- 既存 description line がない状態 → state declaration の直後に新規挿入

### v1.1.0 では やらない (out)

- **State 図全体の hover-insert UX 設計** — 削除のみ、 代替 UX (graph-aware insertion) は v1.x+ で別途検討
- **2 段以上の composite ネスト** — PlantUML の制約 (1 段のみ可)
- **concurrent region (`||`)** — 構文サポートのみ将来検討
- **internal transition** — `state X : trigger [guard] / action @internal` の特殊構文、 v1.x+
- **transition reorder (drag)** — 順序入れ替えは v1.x+
- **buildOverlay の Activity composite 内 rect 制限** — 別 sprint で別途
- **State の同型 mid-insert UX 復活** — graph-aware な代替 UX が必要、 設計負荷大、 別 sprint
- **+ Note 機能** — v1.0.0 で既に実装済、 changes なし
- **state 階層の循環参照検証** — UI 側で dropdown から候補を除外することで防ぐが、 DSL 直書きで作られた循環は検出せず PlantUML に丸投げ

## 3. アーキテクチャ

### 3.1 `src/modules/state.js` の主な変更

#### 3.1.A capabilities 変更

```javascript
capabilities: {
  overlaySelection: true,
  hoverInsert: false,        // ← v1.1.0: graph layout 不適切のため除去
  participantDrag: false,
  showInsertForm: false,     // ← v1.1.0: 同上
  multiSelectConnect: false,
}
```

#### 3.1.B 新規 API (約 8 個)

```javascript
// Transition
function updateTransitionEndpoints(text, lineNum, newFrom, newTo) { ... }

// Composite ops (4)
function convertToComposite(text, stateId) { ... }
function dissolveComposite(text, compositeId) { ... }
function moveStateIntoComposite(text, stateId, targetCompositeId) { ... }
function moveStateOutOfComposite(text, stateId) { ... }

// Behaviors (entry/do/exit)
function setStateBehavior(text, stateId, kind, value) { ... }
function _findStateBlock(lines, stateId) { ... }  // 内部ヘルパ
function _serializeBehaviors(stateId, entry, doVal, exit, indent) { ... }  // 内部ヘルパ
```

すべて export object に追加。

#### 3.1.C parser 拡張

state node の新フィールド:
```javascript
{
  kind: 'state' | 'composite-state',
  id: 'X',
  label: '...',
  stereotype: '...',
  parentId: 'C' | null,        // 既存
  entry: 'start_engine()',     // ← NEW
  do: 'monitor()\nlog()',      // ← NEW (多行は \n escape のまま)
  exit: 'stop_engine()',       // ← NEW
  descriptions: ['custom note 1', 'custom note 2'],  // ← NEW (prefix なし)
  line: 5,
  endLine: 8,
}
```

description line を全て scan、 entry/exit/do prefix で振り分け、 prefix なしは `descriptions` に集約。

#### 3.1.D `_renderStateEdit` / `_renderCompositeStateEdit` 拡張

各 form に以下を追加:

1. **Behaviors セクション** (StableState 準拠の 3 フィールド)
2. **+ Outgoing transition ボタン** (modal 開く)
3. **Move into: dropdown + Move ボタン** (top-level state のみ)
4. **↑ Move out of [parent] ボタン** (parentId 非 null の state のみ)
5. **+ Convert to composite ボタン** (kind === 'state' のみ)
6. **✕ Dissolve composite ボタン** (kind === 'composite-state' のみ)

レイアウト:
```
[State] id / label / stereotype
[Behaviors] entry / do / exit (StableState 流)
[Composite ops]  + Convert / ✕ Dissolve / Move into / Move out (kind と parentId に応じて表示)
[Transition]  + Outgoing transition
[更新] [✕ 構造ごと削除]
```

#### 3.1.E `_renderTransitionEdit` 拡張

既存の trigger / guard / action に加えて:
- **From state**: select dropdown (state IDs)
- **To state**: select dropdown (state IDs)

更新時、 from/to 変更を検出して `updateTransitionEndpoints` を呼ぶ。 trigger/guard/action 変更とは独立 (両方変えてもいい)。

### 3.2 `plantuml-assist.html` の変更

- `#st-tx-modal` (transition 追加用 modal) 新規追加 — `#act-modal` 流用も検討するが、 State 専用に独立させた方が交差汚染しないため別 div
- 既存 `#st-modal` (insert 用) は表示されないため据え置き、 将来削除候補

### 3.3 indent 戦略

- composite 内 state の indent: 親 composite 行の indent + 2 spaces (Activity v1.0.1 の `_resolveInsertIndent` パターンを移植)
- description line (entry/exit/do/custom) の indent: state declaration と同じ indent (composite 内なら 2 spaces)

## 4. ユースケーストレース (use-case-driven-ui-spec)

### UC-1: 単純 state を composite 化して中に既存 state を取り込む (典型シナリオ)

1. ユーザは下の DSL を編集中:
   ```
   state Idle
   state Active
   Idle --> Active : start
   ```
2. `Active` を select → property panel
3. `+ Convert to composite` クリック → DSL: `state Active { }` (空 composite に)
4. `Idle` を select → `Move into:` dropdown で `Active` を選択 → `Move` クリック
5. DSL: `state Active { state Idle }` + `Idle --> Active : start` (transition 据え置き)
6. レンダリング: Active が container、 中に Idle、 外から Active への矢印

### UC-2: composite を解体する (リファクタ)

1. 既存 DSL:
   ```
   state Driving {
     state Slow
     state Fast
     Slow --> Fast : speed_up
   }
   state Stopped
   ```
2. `Driving` を select → `✕ Dissolve composite` クリック → 確認 dialog → OK
3. DSL: `state Slow\nstate Fast\nSlow --> Fast : speed_up\nstate Stopped` (Driving の枠が消え、 内側が top-level に)

### UC-3: state に entry / do / exit を追加 (StableState 流)

1. ユーザは `Driving` を select → property panel の Behaviors セクション
2. entry: `start_engine()`、 do: `monitor_speed()\nlog_telemetry()` (textarea で 2 行入力)、 exit: `stop_engine()` を入力
3. `更新` クリック
4. DSL に description lines が emit:
   ```
   state Driving
   Driving : entry / start_engine()
   Driving : do / monitor_speed()\nlog_telemetry()
   Driving : exit / stop_engine()
   ```
5. レンダリング: state ボックス内に entry/do/exit 行が縦並び表示 (PlantUML が `\n` escape を改行表示)

### UC-4: transition の target を変更する

1. 既存 DSL: `Idle --> Active : start`
2. transition `Idle --> Active` を select (overlay click)
3. property panel: From=`Idle`, To=`Active` (dropdown でいずれも選択可能)
4. `To` を `Stopped` に変更 → `更新` クリック
5. DSL: `Idle --> Stopped : start` (label 維持、 経路だけ変わる)

### UC-5: + Outgoing transition で新規遷移を追加

1. `Idle` を select → property panel
2. `+ Outgoing transition` クリック → modal:
   - Target: `Active` (dropdown)
   - Trigger: `start`
   - Guard: (空)
   - Action: `init`
3. 確定 → DSL に追加:
   ```
   Idle --> Active : start / init
   ```

### UC-6: composite 内 state を top-level に取り出す

1. 既存 DSL: `state C { state X }`
2. `X` を select → property panel に `↑ Move out of C` ボタン表示
3. クリック → DSL: `state X\nstate C { }` (X が top-level、 C は空 composite に)

### UC-7: 既存 state の Behaviors を編集

1. 既存 DSL:
   ```
   state Driving
   Driving : entry / start_engine()
   Driving : do / monitor()
   ```
2. `Driving` を select → Behaviors セクションに既存値が入っている (entry: `start_engine()`、 do: `monitor()`、 exit: 空)
3. exit に `stop_engine()` を入力 → `更新` クリック
4. DSL: 既存 entry/do は据え置き、 新たに `Driving : exit / stop_engine()` 行が追加される

### UC-8: hover-insert が State では発火しないことの確認 (回帰防止)

1. `plantuml-state` を選択
2. 空白部分にホバー
3. `.hover-guide` が **表示されない** (capabilities.hoverInsert が false)
4. 空白クリックでも modal は **開かない** (capabilities.showInsertForm が false)

## 7. リスクと回避

| リスク | 影響 | 回避策 |
|---|---|---|
| description line parser が既存 note と衝突 (note も `STATE : text` 形式に見える) | parse 異常 | note は `note ... of TARGET` 専用構文で parse 済み、 description line は別レーン。 既存 parser test を回帰確認 |
| composite 移動時に内部 transition の参照が宙ぶらりんになる | DSL 不整合 | transition は top-level に残す方針 (PlantUML 越境可)。 削除はしない |
| Move into で循環参照 (X が C の中、 C を X の中に move したい) | 不正 DSL | dropdown 候補から自分自身および descendants を除外 |
| `\n` escape の do 値を textarea で改行表示するときの双方向変換ミス | ユーザの入力値が保存後に変わる | 1 ユニットテストで「textarea 入力 → DSL emit → 再 parse → textarea 表示」 round-trip を保証 |
| composite 内 indent 計算ミス (1 段なら +2 spaces、 多段ならさらに +2) | 視覚的に崩れる、 ただし PlantUML 自体は tolerant | Activity v1.0.1 の `_resolveInsertIndent` パターンを移植し、 親 line の indent + 2 spaces で統一 |
| `+ Outgoing transition` modal で target dropdown が空 (state が 1 個しかない) | self-transition も候補に含めれば回避可 | 自分自身も dropdown 候補に含める (PlantUML は `X --> X` を許容) |

## 9. v1.1.x+ への明示繰越

- **graph-aware な代替挿入 UX** (drag-to-create / connection mode 等) — hoverInsert 除去後の穴埋め
- **2 段以上の composite ネスト** (PlantUML 制約のため後回し)
- **concurrent region** (`||`)
